## 🔴 Hallazgos CRÍTICOS

### C1. SQL injection en endpoints ABM de servicios

**Dónde:** workflow `💳 API Pagos Servicios`, nodos `Crear Servicio DB`, `Actualizar Servicio DB`.

**Problema:** la query embebe directamente el JSON del body con `'{{ $json.nombre }}'`. El Code node hace `replace(/'/g, "''")` para escapar comillas simples pero NO escapa:
- Backslashes (`\`) → puede romper escape de Postgres
- Numeric fields (`monto_estimado`, `porcentaje_propio`) → un payload con `"monto_estimado": "0; DELETE FROM servicios_recurrentes;--"` atravesa la validación si parseFloat no falla en parte de la cadena

**Impacto:** quien tenga la URL del webhook puede ejecutar SQL arbitrario contra `finanzas_personales`.

**Mitigación inmediata:**
1. Usar parámetros posicionales `$1, $2` con `queryReplacement` (postgres v2.6 lo soporta).
2. En Code node, validar tipos estrictos: `parseFloat(x)` debe ser finito Y >= 0 Y < 1e9; rechazar si no.
3. Whitelist de valores enum: `tipo` ∈ {servicio, impuesto, suscripcion, alquiler, seguro, otro}, `moneda` ∈ {ARS, USD}, etc.

**Workaround temporal:** poner el webhook detrás de un token (ver C2).

---

### C2. Webhooks sin autenticación

**Dónde:** todos los `/webhook/*` (pagar-servicio, desmarcar-pago, dash-data, crear/actualizar/eliminar-servicio, asistente-financiero, router-wa).

**Problema:** cualquiera con la URL puede invocarlos. Si alguien hace `scrape` de tu PWA, ve la URL del API y puede:
- Marcar/desmarcar pagos (toca billeteras)
- Crear/eliminar servicios
- Consumir tu cuota de OpenAI (asistente-financiero)

**Mitigación:**
1. **Token bearer simple** (10 min de trabajo): generar token random, agregarlo como header `X-API-Key` en todos los webhooks excepto `router-wa` (Evolution API). Validar en Code node antes de ejecutar SQL.
2. Almacenar token en variable `n8n.webhookToken` (env var de n8n) y en `localStorage` del PWA (acepto que es client-side; al menos eleva la barrera).
3. **Mejor opción** (más trabajo): Cloudflare Worker o JWT con expiración corta firmado por backend.

---

### C3. Webhooks sin `onError: continueRegularOutput`

**Dónde:** los 5 webhooks del workflow API Pagos.

**Problema:** si el postgres falla (constraint violation, conexión caída), el RespondToWebhook node no se ejecuta → el cliente HTTP queda esperando hasta timeout (60s default) → en el PWA aparece "Error al registrar pago" sin detalle.

**Mitigación:** patchear los 5 webhooks para agregar `parameters.options.responseHeaders` y `onError: "continueRegularOutput"` al postgres node, con un Set node de error fallback que arme un response con `ok:false, error:"<mensaje>"`.

## 🟠 Hallazgos ALTOS

### H1. Manejo de timezone en `$now`

**Problema:** `$now` en n8n usa la TZ del servidor. Tu VPS (Contabo) probablemente está en UTC. Si registrás un gasto a las 23:30 hora Argentina, `$now.toFormat('yyyy-MM-dd')` puede devolver el día siguiente porque internamente son 02:30 UTC.

**Verificación rápida:**
```bash
docker exec <n8n-container> date    # ver TZ del servidor n8n
SELECT NOW(), NOW() AT TIME ZONE 'America/Argentina/Buenos_Aires';
```

**Mitigación:** en el system prompt y en queries críticas usar `{{ $now.setZone('America/Argentina/Buenos_Aires').toFormat('yyyy-MM-dd') }}`. Confirmé que Luxon soporta `.setZone()`.

---

### H2. Manejo de USD inconsistente

**Problemas identificados:**

1. **Pago de servicio USD no descuenta de ninguna billetera.** El CTE de `pagar-servicio` tiene `AND pago_upsert.moneda = 'ARS'` en el UPDATE billeteras → si pagás Claude Pro (USD 20) en efectivo, queda registrado el gasto pero la billetera USD no se ve afectada. Luis paga USD desde "Fondo para Auto" o "Dinero Operativo USD" pero el sistema no lo refleja.

2. **`monto_real` se guarda en moneda nativa pero `gastos_variables.monto` se convierte a ARS.** Bien, pero la conversión usa `dolar_blue` del momento del pago, no de la fecha real del cargo. Si el TC sube/baja entre el día del pago y el día del registro, hay drift.

3. **Modal de pago acepta input numérico genérico** sin validar que coincida con la moneda del servicio.

**Mitigación:**
1. Agregar lógica al CTE: si `moneda = 'USD'`, descontar de billetera USD (Fondo para Auto u otra explícita configurable).
2. Documentar en la tabla `servicios_recurrentes` qué billetera específica afecta cada servicio (campo `billetera_origen_id`).

### H5. No idempotency keys en webhooks de pago

**Problema:** si la red del cliente PWA tiene flap, el `fetch` puede reintentar internamente y llegar 2x al backend. La defensa actual es `ON CONFLICT (servicio_id, periodo)` que evita duplicar el pago, pero el `gasto_ins` tiene `WHERE NOT EXISTS` — está OK. El `bil_upd` se ejecutaría 2 veces si el ON CONFLICT dispara UPDATE (que reusa monto_real).

**Mitigación:** agregar idempotency-key en header `X-Idempotency-Key: <uuid>` y guardarlo en una tabla `idempotency_keys (key, response_json, created_at)` con TTL de 24h. Si llega un request con key conocido, devolver respuesta cacheada.

---

### H6. Agente sin confirmación en operaciones destructivas

**Problema:** el system prompt dice "Ejecutar DIRECTO con descripcion + monto + metodo" para `registrar_gasto`. Si el usuario manda *"borrá el último gasto"*, el agente puede invocar `eliminar_gasto` sin doble check.

**Mitigación:** agregar al prompt:
```
### Operaciones destructivas (SIEMPRE confirmar)
- eliminar_gasto, modificar_gasto (cambio mayor a 30%), desmarcar_pago_servicio
- Workflow: "Confirmá: vas a [acción]. Respondé SI para ejecutar."
- Solo ejecutar si la respuesta del usuario es claramente afirmativa.
```

## 🟡 Hallazgos MEDIOS

### M1. Hardcoded YCloud number `5493815762656`

5+ nodos lo tienen hardcoded. Si querés probar otro número (sandbox, segundo número), tenés que editar todos. Mover a credential metadata o env var.

### M2. Sin auditoría de cambios

Cuando un servicio se actualiza/elimina, no se registra quién/cuándo/qué cambió. Misma cosa para pagos.

**Mitigación:** tabla `audit_log (id, table_name, record_id, action, old_values jsonb, new_values jsonb, source, timestamp)` poblada por triggers en cada tabla crítica.

### M3. Categorización por ILIKE es frágil

```sql
WHERE nombre ILIKE '%servicio%' OR nombre ILIKE '%fijo%'
```
Si renombrás la categoría a "Cuentas mensuales", deja de matchear → los pagos de servicio se asignan a NULL → constraint violation.

**Mitigación:** referencia por ID (configurable en una tabla `system_config`).

### M4. Servicios eliminados no se pueden reactivar desde la UI

El soft delete pone `activo = false`. No hay UI para verlos ni reactivarlos. Si querés recuperar uno, SQL manual.

**Mitigación:** toggle "Mostrar inactivos" en la barra de filtros del tab Servicios.

### M5. No hay backup verificado del schema

Supabase hace daily backup automático en el plan free pero solo retiene 1 día. Si te equivocás un viernes y notás el lunes, perdiste.

**Mitigación:**
1. `pg_dump` semanal a Storage de Supabase o GitHub privado.
2. Documentar el procedimiento de restore.

### M6. Sin healthcheck del Motor Financiero

El Motor corre cron diario 09:00. Si falla (workflow desactivado, postgres caído, OpenAI rate limit), nadie se entera hasta que faltan datos.

**Mitigación:** agregar al Motor un nodo final que mande `POST` a un dead-man's-switch (tipo healthchecks.io free) o un mensaje al WhatsApp de Luis "✅ Motor OK [fecha]".

### M7. Falta validación de `dia_vencimiento` 1-31

El Code node hace `parseInt` pero no valida rango. Si querés guardar `dia_vencimiento = 35`, queda en BD y al renderizar puede dar fechas inválidas.

### M8. PWA sin manejo de offline real

Service worker cachea el shell pero las operaciones (pagar, crear servicio) requieren red. Si abrís el PWA sin conexión, podés ver el último estado pero no podés operar y los errores son crípticos.

**Mitigación:** queue de operaciones offline + Background Sync API.

## 🟢 Hallazgos BAJOS

### L1. Service worker sin estrategia de update
Cada release requiere bumpear `CACHE = 'finanzas-vN'`. Olvido fácil → usuario ve versión vieja. Considerar `self.registration.update()` periódico.

### L2. Modal pago no soporta tab Enter para pasar entre fields
Solo el monto envía con Enter. UX pequeño.

### L3. KPI "Total mensual" en Servicios no respeta filtros
Está intencional (mostrar totales globales) pero algunos usuarios esperan que sea reactivo. Documentarlo o hacer toggleable.

### L4. Logs de n8n no rotan
Crecen indefinidamente. Configurar `EXECUTIONS_DATA_PRUNE=true` y `EXECUTIONS_DATA_MAX_AGE=336` (14 días).
